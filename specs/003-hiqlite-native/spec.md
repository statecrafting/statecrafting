# 003: In-process hiqlite native addon

## 1. Purpose

hiqlite-native is the second-deepest chassis dependency after the toolchain
(spec 002): every stamped app that uses in-process cache/KV or counters
loads it, and so does statecraft. It was named `@enrahitu/hiqlite-native`,
the same scope-lies-about-ownership case as the toolchain (spec 001
section 1). It moves here to sit beside the toolchain under one scope.

## 2. Territory

A single napi-rs package, `addon/hiqlite-native/`, plus the per-platform
binary packages napi generates from it:

| Package | Contents |
|---|---|
| `@statecrafting/hiqlite-native` | the napi loader (`index.js` + `index.d.ts`, built); no binary |
| `@statecrafting/hiqlite-native-darwin-arm64` | the `.node` for macOS arm64 |
| `@statecrafting/hiqlite-native-linux-x64-gnu` | the `.node` for linux x64 |
| `@statecrafting/hiqlite-native-linux-arm64-gnu` | the `.node` for linux arm64 |

The Rust crate (`Cargo.toml`, `build.rs`, `src/lib.rs`) is self-contained:
it depends on upstream `hiqlite = 0.14` from crates.io and the napi
runtime, with no workspace or path dependency, so the move is a clean copy
of the tracked source. `index.js`, `index.d.ts`, and the `.node` binaries
are build output (`napi build`) and are gitignored; the platform package
directories under `addon/hiqlite-native/npm/` are generated at publish time
by `napi create-npm-dirs` and are likewise not committed.

## 3. Behavior

### License

Apache-2.0. hiqlite-native is a thin wrapper we wrote over upstream hiqlite
(itself Apache-2.0 / MIT), and it is consumed by stamped customer apps, so
it belongs to the permissive tier (spec 001 section 3). It carries its own
Apache-2.0 LICENSE file. It must not depend on `governance-native` or
`fleet-native` (spec 001 section 3.2).

### Publishing

Joins the spec 002 publish matrix (`.github/workflows/publish.yml`): each
platform leg runs `napi build` and publishes its `.node` package; one leg
publishes the meta package. Idempotent per version, same as the toolchain.
The binaries are built natively per platform, so publishing is CI-only.

## 4. Ownership transfer

Per spec 001 section 4: this drops the `addon/` edge from
enrahitu/002-in-process-hiqlite, which keeps its other edge
(`backend/hiq/`, the Encore-side hiq service) and stays `approved` /
`complete`. No spec retires.

## 5. Acceptance

1. `@statecrafting/hiqlite-native` and its three platform packages publish
   at `0.1.0` with provenance, all declaring Apache-2.0 with a LICENSE
   file.
2. enrahitu builds against `@statecrafting/hiqlite-native@0.1.0` with
   `addon/` removed from its tree, and its gates pass. **Satisfied
   2026-07-22** (statecrafting/enrahitu PR #25): `addon/` is deleted,
   `backend/hiq/init.ts` imports `@statecrafting/hiqlite-native`, and the
   `verify` suite passes in CI (the addon crate is a byte-identical move, so
   the runtime behavior is unchanged).
3. statecraft builds against `@statecrafting/hiqlite-native@0.1.0` and its
   gates pass. **Satisfied 2026-07-22** (statecrafting/statecraft PR #54).
4. enrahitu/002 no longer claims `addon/`; it still owns `backend/hiq/`.
   **Satisfied** (enrahitu PR #25): the `addon/` edge drops from enrahitu spec
   002-in-process-hiqlite, which keeps `backend/hiq/`.
5. **Operator step:** deprecate `@enrahitu/hiqlite-native*` on npm with a
   message naming the new package (needs an `@enrahitu`-scoped token).

## 6. Out of scope

- Bumping upstream hiqlite past 0.14; this move preserves the pin.
- The Encore-side `backend/hiq/` service, which stays in enrahitu.
